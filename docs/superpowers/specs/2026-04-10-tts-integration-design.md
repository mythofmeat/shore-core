# TTS Integration Design

**Date:** 2026-04-10
**Status:** Draft

## Overview

Add text-to-speech support to Shore, enabling characters to speak their responses aloud. Audio is generated by an external TTS server (ttsd) exposing an OpenAI-compatible API, proxied through the daemon, and played back on the client.

## Topology

```
client (shore-cli/tui)  ‚îÄ‚îÄSWP‚îÄ‚îÄ>  daemon (shore-daemon)  ‚îÄ‚îÄHTTP‚îÄ‚îÄ>  ttsd (GPU box)
     rodio playback                  proxy/relay                    audio generation
```

- Daemon proxies TTS requests and relays audio bytes ‚Äî never plays audio itself
- Client handles all audio playback locally via rodio/cpal
- ttsd runs on a separate machine with GPU access

## Two Modes

1. **On-demand**: User requests speech of a specific message via `shore speak [ref]` or `:speak [ref]`
2. **Live mode**: Toggled via `shore speak on/off` or `:speak on/off`. Every assistant response is automatically spoken as it completes.

## SWP Protocol Additions

### Client ‚Üí Server

```rust
ClientMessage::Speak {
    rid: Option<String>,
    msg_ref: Option<String>,  // existing ref system, None = last assistant msg
}

ClientMessage::SetLiveSpeak {
    rid: Option<String>,
    enabled: bool,
}
```

### Server ‚Üí Client

```rust
ServerMessage::AudioStart {
    rid: Option<String>,
    msg_id: String,
    sample_rate: u32,
    channels: u16,
}

ServerMessage::AudioChunk {
    rid: Option<String>,
    data: String,  // base64-encoded WAV bytes
}

ServerMessage::AudioEnd {
    rid: Option<String>,
}

ServerMessage::AudioError {
    rid: Option<String>,
    message: String,
}
```

### Sequencing

**On-demand:** Client sends `Speak` ‚Üí server responds with `AudioStart ‚Üí AudioChunk* ‚Üí AudioEnd` (or `AudioError`).

**Live mode:** Client sends `SetLiveSpeak { enabled: true }`. During subsequent generations, after `StreamEnd`, the daemon automatically issues `AudioStart ‚Üí AudioChunk* ‚Üí AudioEnd` to that client using the generated response text.

## Daemon: TTS Proxy Module

New module: `shore-daemon/src/tts/` (~200-300 LOC)

### client.rs

HTTP client for ttsd's OpenAI-compatible API.

- `speak(text: &str, voice: &str) -> Result<impl AsyncRead>` ‚Äî `POST /v1/audio/speech` with streaming response
- Request body: `{"model": "", "input": "<text>", "voice": "<character_name>"}`
- Response: streaming `audio/wav` bytes
- Uses reqwest with streaming (no full-body buffering)

### config.rs

```rust
struct TtsConfig {
    enabled: bool,
    host: String,
    port: u16,
}
```

### relay.rs

Bridges the HTTP audio stream to SWP messages.

- Reads WAV bytes from reqwest response in ~8KB chunks
- Base64-encodes each chunk into `AudioChunk` messages
- Sends over the requesting client's SWP connection
- On error (ttsd down, voice not found): sends `AudioError`

### Handler Integration

- `Speak` messages: Handler resolves msg ref ‚Üí gets text content ‚Üí spawns async task ‚Üí calls `client.speak()` ‚Üí relay streams back
- Live mode: After `StreamEnd` in `generation.rs`, if requesting client has live mode enabled, handler triggers the same speak flow with the response text
- Live mode state: Per-connection flag in the server's client registry

## Client: Audio Playback

New module in `shore-client`: `audio.rs` (~150 LOC)

### AudioPlayer

```rust
struct AudioPlayer {
    stream: Option<OutputStream>,
    sink: Option<Sink>,
    buffer: AudioBuffer,  // ring buffer, SPSC
}
```

- `AudioStart`: Create rodio `OutputStream` + `Sink`, configure sample rate/channels
- `AudioChunk`: Decode base64, append WAV bytes to ring buffer. Rodio drains from the other end ‚Äî playback begins on first chunk
- `AudioEnd`: Signal buffer EOF, rodio drains remainder
- `AudioError`: Drop sink, surface error to user

Ring buffer bridges async SWP receiver and rodio's synchronous audio thread. Custom `Source` trait impl reads from buffer, outputs silence on momentary underrun.

### Interruption

- CLI: Ctrl+C (normal signal handling)
- TUI: Esc or `:speak stop` ‚Äî drops the Sink, immediate stop

### Build Dependencies

- `rodio` and `cpal` added to `shore-client/Cargo.toml`
- Compile-time: `alsa-lib` headers (Linux)
- Runtime: `libasound.so` (present on any Linux system with audio)

## CLI Commands

`shore speak` subcommand:

- `shore speak` ‚Äî speak last assistant message
- `shore speak <ref>` ‚Äî speak referenced message (existing ref resolution)
- `shore speak on` ‚Äî enable live mode for this session
- `shore speak off` ‚Äî disable live mode

On-demand speak: connects via SWP, sends `Speak`, receives audio messages, feeds `AudioPlayer`, exits after `AudioEnd`.

Live mode during `shore send`: audio messages handled in the existing streaming receive loop, interleaved with `StreamChunk` messages.

## TUI Commands

`:speak` in the command bar:

- `:speak` ‚Äî speak last assistant message
- `:speak <ref>` ‚Äî speak referenced message
- `:speak on` / `:speak off` ‚Äî toggle live mode
- `:speak stop` ‚Äî stop current playback

Status bar indicator: `[TTS]` shown when live mode is enabled.

Audio messages handled in the TUI's main event loop. `AudioPlayer` runs on a background thread ‚Äî TUI stays responsive.

## Configuration

Global config (`~/.config/shore/config.toml`):

```toml
[tts]
enabled = false
host = "192.168.1.50"
port = 8778
```

- `Option<TtsConfig>` field on `AppConfig`
- Section absent ‚Üí TTS unavailable
- `enabled = false` ‚Üí TTS commands return clear error
- No per-character config ‚Äî voice name = character name by convention
- No backend/param config ‚Äî managed entirely by ttsd

## Voice Convention

- Voice name = character name. When speaking for character `morgan`, daemon sends `"voice": "morgan"` to ttsd.
- Voice files (`.wav`, `.meta.toml`) live in ttsd's `--voices-dir`, not in Shore.
- Shore does not manage, upload, or reference voice files.
- If ttsd doesn't have a matching voice, it returns an error which Shore surfaces to the user.

## Crate Boundaries

| Crate | Changes |
|-------|---------|
| `shore-protocol` | New `ClientMessage` and `ServerMessage` variants |
| `shore-config` | New `TtsConfig` struct, `[tts]` section parsing |
| `shore-daemon` | New `tts/` module (client, relay, config), handler integration |
| `shore-client` | New `audio.rs` (AudioPlayer, rodio integration) |
| `shore-cli` | New `speak` subcommand |
| `shore-tui` | New `:speak` command, status bar indicator |

## Out of Scope

- Voice management (upload, listing) ‚Äî lives in ttsd
- Per-character voice overrides ‚Äî not needed, convention is sufficient
- Audio caching ‚Äî not needed for MVP
- Reference text sidecars ‚Äî not needed for current backend
- Multiple TTS provider backends ‚Äî design supports swapping ttsd URL for ElevenLabs/OpenAI later, but only ttsd is implemented now
