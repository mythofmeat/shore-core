## Features

- **Token-based compaction trigger.** Current `[memory.compaction] max_turns` is count-based, a poor proxy for cost because per-turn content varies wildly (heavy-thinking turns = 2–3K tokens, light chat turns = 400). Add a `max_context_tokens` knob that triggers compaction when the estimated prompt token count exceeds threshold, regardless of turn count. Keep `min_turns` as a floor to prevent thrash. Empirically derived from 30-day ledger simulation: the per-call cost curve has an elbow at ~30K context, where median cost jumps from ~$0.03 to ~$0.08 on Opus 4.7. Setting threshold at 30,000 tokens saves ~3.3% on typical days and up to ~8% on heavy-thinking spike days. Below 30K: savings don't outrun the extra compaction cost. Above 30K: leave fat-tail money on the table. Empirical source: simulate against ledger CSV, bucket calls by (input+cache_read+cache_write), walk cycles and score hypothetical compactions — run the simulation again when Opus pricing or conversation patterns shift. Principled but specific to this character/model; document both.

## Bugs

- **Ledger `thinking_enabled` column misses `reasoning_effort`.** In `shore-daemon/src/handler/generation.rs:223-227` the flag is derived solely from `provider_options.budget_tokens`. The OpenAI-style `reasoning_effort` knob (used by our current Opus 4.7 + OpenRouter config) takes a different code path and never sets `budget_tokens`, so `thinking_enabled` is recorded as `false` for every call even when reasoning tokens come back. Makes cost debugging actively misleading — e.g. Apr 17 2026 had 18,608 reasoning tokens across 87 calls, all logged as `thinking_enabled=false`. Fix: set true when either `budget_tokens` OR `reasoning_effort` is present in provider_options.

- **Thinking blocks from completed prior turns are re-sent to the API on every subsequent call.** `shore-daemon/src/content_util.rs:14-22` (`content_block_to_api_json`) unconditionally serializes signed `ContentBlock::Thinking` into request JSON. Callers at `engine/tools.rs:115`, `handler/persistence.rs:91`, and `autonomy/manager.rs:1433` all apply it to full history. Anthropic's extended-thinking semantics require thinking blocks to persist *within* an in-progress tool-use loop (same turn) but they serve no purpose after `end_turn` — the model does not attend to prior-turn thinking (confirmed by live testing on 2026-04-17). Measured cost on Apr 17 2026: ~$1.12 of pure waste from re-sending 10.7K accumulated thinking tokens across 94 calls (cache reads + fresh input + cache writes). Fix: at the history-serialization level, strip `Thinking`/`RedactedThinking` blocks from assistant messages that are *not* part of the in-progress tool-use chain. `prompt.rs:662 is_tool_loop_msg_prompt` already distinguishes tool-loop intermediates; reuse that signal. **This behavior should be configurable** — a `[memory.thinking] preserve_prior_turns = false` toggle (default false) would let users opt back in if a future use case wants the old behavior, but the default should save money.
